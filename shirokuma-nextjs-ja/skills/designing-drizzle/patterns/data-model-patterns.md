# データモデルパターン

Drizzle ORM データモデル設計判断のためのパターン比較テーブル。

## テーブル設計

### 主キー戦略

| パターン | 使用場面 | トレードオフ |
|---------|---------|------------|
| UUID (`uuid().defaultRandom()`) | 分散システム、公開 ID | ランダムでインデックス効率が低い |
| CUID2 (`text().$default(createId)`) | URL-safe な公開 ID | UUID より短い、DB ネイティブ型ではない |
| 連番 (`serial()`) | 内部 ID、小規模テーブル | 予測可能、分散に不向き |
| UUID v7 (`uuid().$default(uuidv7)`) | 時系列ソート + ユニーク性 | 最新の標準、ライブラリ追加が必要 |

### 判定: UUID vs 連番

| 基準 | UUID | 連番 |
|------|------|------|
| 公開 API で ID を露出する | 推奨 | セキュリティリスク |
| 挿入順でのソート | UUID v7 を使用 | 自然にソート |
| 外部キーの結合コスト | やや高い（16 bytes） | 低い（4 bytes） |
| マルチテナント / 分散 | 推奨 | 衝突リスク |
| 単純な管理画面向けテーブル | オーバーヘッド | 推奨 |

### 共通カラムパターン

| パターン | カラム | 用途 |
|---------|--------|------|
| タイムスタンプ | `createdAt`, `updatedAt` | 監査証跡 |
| ソフトデリート | `deletedAt` | 論理削除 |
| バージョニング | `version` | 楽観的ロック |
| テナント分離 | `tenantId` | マルチテナント |
| 作成者追跡 | `createdBy`, `updatedBy` | 操作者の記録 |

### 命名規約

| 要素 | 規約 | 例 |
|------|------|-----|
| テーブル名 | snake_case、複数形 | `users`, `blog_posts` |
| カラム名 | camelCase（Drizzle 標準） | `createdAt`, `userId` |
| 外部キー | `{entity}Id` | `userId`, `postId` |
| ジャンクションテーブル | `{entity1}_to_{entity2}` | `users_to_roles` |
| インデックス | `{table}_{columns}_idx` | `posts_author_id_idx` |
| ユニーク制約 | `{table}_{columns}_unique` | `users_email_unique` |

## リレーション設計

### リレーション型

| パターン | Drizzle 定義 | 使用場面 |
|---------|-------------|---------|
| One-to-One | `one(target, { fields, references })` | ユーザー ↔ プロフィール |
| One-to-Many | `one()` + `many()` | 著者 → 記事 |
| Many-to-Many | ジャンクションテーブル + `many()` x2 | ユーザー ↔ ロール |
| 自己参照 | 同テーブルへの `one()` | カテゴリの親子関係 |

### ON DELETE 戦略

| 動作 | 使用場面 | リスク |
|------|---------|--------|
| `CASCADE` | 親と常にライフサイクルが同じ子 | 意図しない大量削除 |
| `SET NULL` | 参照先が削除されても残すべきデータ | NULL チェックが必要 |
| `RESTRICT` | 子が存在する場合の削除を禁止 | 操作エラーになる |
| `NO ACTION` | アプリケーション側で制御 | 孤児レコードリスク |

### 判定: CASCADE vs SET NULL vs RESTRICT

| 基準 | CASCADE | SET NULL | RESTRICT |
|------|---------|----------|----------|
| コメント → 投稿（投稿削除時） | 推奨 | — | — |
| 注文 → ユーザー（ユーザー削除時） | 危険 | 推奨 | 推奨 |
| プロフィール → ユーザー（1:1） | 推奨 | — | — |
| タグ付け → タグ（タグ削除時） | 推奨 | — | — |
| 請求書 → 顧客（法的保持） | 禁止 | 推奨 | 推奨 |

### クエリ結合パターン

| パターン | 使用場面 | Drizzle API |
|---------|---------|------------|
| リレーショナルクエリ | ネスト結果が必要 | `db.query.users.findMany({ with: { posts: true } })` |
| SQL 結合 | 集計、複雑な条件 | `db.select().from(users).leftJoin(posts, eq(...))` |
| サブクエリ | EXISTS / NOT EXISTS チェック | `db.select().from(users).where(exists(...))` |

## インデックス戦略

### インデックス種別

| 種別 | 使用場面 | Drizzle 定義 |
|------|---------|-------------|
| 単一カラム | 等値検索、外部キー | `index('idx_name').on(table.column)` |
| 複合 | 複数条件の WHERE | `index('idx_name').on(table.col1, table.col2)` |
| ユニーク | 一意制約 | `uniqueIndex('idx_name').on(table.column)` |
| 部分 | 条件付きインデックス（PostgreSQL） | `index('idx_name').on(table.column).where(sql`...`)` |

### インデックス設計ルール

| ルール | 説明 |
|--------|------|
| 外部キーには常にインデックス | JOIN パフォーマンスの基本 |
| WHERE 句の頻出カラム | 検索条件に合わせる |
| 複合は選択性の高い順 | カーディナリティの高いカラムを先頭に |
| カバリングインデックスを検討 | SELECT 対象も含める（読み取り最適化） |
| 書き込み頻度が高いテーブルは控えめに | INSERT/UPDATE のオーバーヘッド |

### 判定: インデックス追加の基準

| 基準 | インデックスを追加 | インデックス不要 |
|------|------------------|----------------|
| 外部キーカラム | 常に | — |
| WHERE 句で頻繁に使用 | 推奨 | — |
| テーブル行数 < 1000 | — | フルスキャンで十分 |
| 書き込み > 読み取り | 慎重に | 過剰なインデックスは避ける |
| LIKE '%keyword%' | — | B-Tree では効果なし（全文検索を検討） |

## ソフトデリート

### 実装パターン

| パターン | カラム | 利点 | 欠点 |
|---------|--------|------|------|
| Nullable timestamp | `deletedAt: timestamp()` | 削除日時が記録される | NULL チェックが必要 |
| Boolean flag | `isDeleted: boolean().default(false)` | シンプル | 削除日時が不明 |
| ステータス enum | `status: statusEnum()` | 複数状態を表現 | 複雑 |

### 推奨: Nullable timestamp

```
deletedAt: timestamp('deleted_at', { mode: 'date' })
```

- `NULL` = アクティブ、非 NULL = 削除済み
- 削除日時が監査証跡として有用
- 部分インデックスと組み合わせ可能（PostgreSQL）

### ソフトデリートの適用判定

| 基準 | ソフトデリート | ハードデリート |
|------|-------------|-------------|
| 法的データ保持要件 | 推奨 | 禁止 |
| ユーザーがゴミ箱から復元 | 推奨 | — |
| 監査証跡が必要 | 推奨 | — |
| 一時データ（セッション等） | — | 推奨 |
| ストレージコスト懸念 | — | 推奨 |
| 外部キーで参照される | 推奨 | CASCADE が必要 |

### クエリパターン

| 操作 | クエリ |
|------|--------|
| アクティブレコード取得 | `.where(isNull(table.deletedAt))` |
| 削除済み含む全件 | フィルタなし |
| 論理削除実行 | `.set({ deletedAt: new Date() })` |
| 復元 | `.set({ deletedAt: null })` |
| 完全削除 | `.delete().where(isNotNull(table.deletedAt))` |

## マイグレーション戦略

### drizzle-kit ワークフロー

| コマンド | 用途 | タイミング |
|---------|------|-----------|
| `drizzle-kit generate` | SQL マイグレーションファイル生成 | スキーマ変更後 |
| `drizzle-kit migrate` | マイグレーション適用 | デプロイ時 |
| `drizzle-kit push` | 直接 DB にスキーマ適用 | 開発中のみ |
| `drizzle-kit studio` | DB ブラウザ起動 | デバッグ時 |

### マイグレーション分類

| 種別 | 例 | リスク | 対策 |
|------|-----|--------|------|
| 加算的 | カラム追加、テーブル追加 | 低 | デフォルト値を設定 |
| 変更的 | カラム型変更、リネーム | 中 | ダウンタイムを考慮 |
| 削除的 | カラム削除、テーブル削除 | 高 | 段階的に実施（非推奨化 → 削除） |
| データ | 既存データの変換 | 高 | スキーマ変更と分離 |

### 安全なマイグレーション手順

1. **加算的変更を先に**: 新カラム/テーブルを追加（デフォルト値付き）
2. **アプリケーションコードを更新**: 新スキーマを使用するよう変更
3. **データ移行**: 必要に応じて既存データを変換
4. **削除的変更を最後に**: 旧カラム/テーブルを削除

### 判定: push vs generate+migrate

| 基準 | `push`（開発向け） | `generate` + `migrate`（本番向け） |
|------|------------------|----------------------------------|
| ローカル開発 | 推奨（高速） | オーバーヘッド |
| ステージング / 本番 | 禁止 | 推奨 |
| チーム開発 | リスク | 推奨（履歴管理） |
| データ保持が必要 | リスク | 推奨 |
