# 汎用アーキテクチャ設計

フレームワーク非依存のアーキテクチャ設計。設計判断とトレードオフ分析に集中し、実装は `code-issue` に委任する。

> **アーキテクチャ設計はこのスキルの責務。** `code-issue` はここで決定された設計に基づいて実装を担当する。

## スコープ

- **カテゴリ:** 調査系ワーカー
- **スコープ:** 技術スタック・既存コード構造の読み取り（Read / Grep / Glob / Bash 読み取り専用コマンド）、アーキテクチャ設計ドキュメントの生成（Write/Edit — 設計成果物への書き込み）、Issue 本文への設計セクション追記
- **対象ドメイン:** CLI ツール、ライブラリ、汎用 TypeScript / Node.js プロジェクト
- **スコープ外:** プロダクションコードの実装（`code-issue` に委任）、ビルド検証、フレームワーク固有設計（Next.js → `designing-nextjs`、データモデル → `designing-drizzle`、UI → `designing-shadcn-ui`）

> **設計成果物の書き込みについて**: このスキルが Issue 本文・設計ドキュメントに Write/Edit するのは設計プロセスの成果物であり、プロダクションコードの変更ではない。調査系ワーカーの例外として許可される。

## ワークフロー

### 0. 技術スタック確認

**最初に**、プロジェクトの `CLAUDE.md` を読んで確認:
- 使用言語・ランタイム（Node.js、Deno 等）とバージョン
- パッケージマネージャー（npm / pnpm / yarn）
- ビルドツール（TypeScript コンパイラ、esbuild、rollup 等）
- 既存のディレクトリ構造とモジュール分割方針
- テストフレームワーク（Jest、Vitest 等）

`.claude/rules/` 内の `tech-stack.md` も確認する。

### 1. 設計コンテキスト確認

`design-flow` から委任された場合、Design Brief と要件が渡される。そのまま使用する。

スタンドアロンで起動された場合、Issue 本文と計画セクションから要件を把握する。

### 2. アーキテクチャ分析

Issue に関連する各設計観点について、決定フレームワークを適用する:

#### 設計観点

| 観点 | 対処するタイミング | 主な検討事項 |
|------|------------------|------------|
| モジュール構成 | 新規モジュール追加、責務分割 | 単一責任・依存方向・公開インターフェース |
| インターフェース設計 | モジュール境界の型定義 | 型安全性・拡張性・後方互換性 |
| 依存性管理 | モジュール間の依存、循環依存 | 依存性注入・ファクトリパターン |
| コマンドパターン | CLI コマンド追加・変更 | コマンド分割・オプション設計・エラーハンドリング |
| 移行設計 | 既存コードのリファクタリング、API 変更 | 段階的移行・後方互換・廃止戦略 |
| ビルドパイプライン | ビルド・バンドル・出力形式 | ESM/CJS 対応・型定義出力・ツリーシェイキング |

#### 決定フレームワーク

各観点について評価する:

1. **要件**: 機能・非機能要件を整理する
2. **制約**: 既存コード構造、Node.js バージョン、依存ライブラリ
3. **選択肢**: 実行可能なアーキテクチャパターンを列挙する
4. **トレードオフ**: 決定マトリクスでオプションを比較する
5. **決定**: 根拠を添えてアーキテクチャを選択する

#### よく使うパターン

| パターン | 適用場面 |
|---------|---------|
| コマンドパターン | CLI コマンドのカプセル化・undo サポート |
| ファクトリパターン | オブジェクト生成の抽象化・テスト容易性向上 |
| リポジトリパターン | データアクセスの抽象化・ストレージ差し替え |
| ストラテジーパターン | アルゴリズム・動作の切り替え |
| コンポジットパターン | ツリー構造・再帰処理 |
| プラグインパターン | 拡張ポイント・動的ロード |
| 依存性注入 | テスト容易性・モジュール間結合の緩和 |

### 3. 設計出力

構造化されたドキュメントとしてアーキテクチャ設計を作成する:

```markdown
## アーキテクチャ設計

### 設計概要
{変更の目的と影響範囲の要約}

### モジュール構成
{ディレクトリ構造またはモジュール依存グラフ}

### インターフェース定義
{主要な型定義・インターフェース（TypeScript 形式）}

### 移行計画
{既存コードからの移行手順（段階ごとに整理）}
（新規設計の場合は省略）

### ビルド・配布への影響
{package.json の変更点、出力形式の変更等}
（影響がない場合は省略）

### 主要決定事項
| 決定 | 選択 | 根拠 |
|------|------|------|
| {トピック} | {パターン} | {理由} |
```

### 4. レビューチェックリスト

- [ ] 単一責任原則 — 各モジュールの責務が明確
- [ ] 依存方向が一方向（循環依存なし）
- [ ] 公開インターフェースが最小限（内部実装を隠蔽）
- [ ] 型定義が型安全（`any` を避ける）
- [ ] 既存テストへの影響範囲を確認
- [ ] 移行が段階的（後方互換を維持、または廃止戦略が明確）
- [ ] 既存コードパターン・命名規約と整合

## リファレンスドキュメント

| ドキュメント | 内容 | 読むタイミング |
|------------|------|--------------|
| `CLAUDE.md` | プロジェクト概要・技術スタック | 設計開始時 |
| `tech-stack.md`（ルール） | 推奨技術スタック | 技術選定時 |
| 既存 `src/` | 現在のコード構造 | パターン整合性確認時 |

## アンチパターン

| パターン | 問題 | 代替案 |
|---------|------|--------|
| 神クラス（God Class） | 責務過多で変更困難 | 単一責任原則に従い分割 |
| 循環依存 | ビルド・テスト・理解が困難 | 依存方向を一方向に整理、共通インターフェースを抽出 |
| 過度な抽象化 | 必要以上の間接参照で可読性低下 | YAGNI 原則 — 実際に必要になるまで抽象化しない |
| 巨大関数 | テスト・変更が困難 | 小さな純粋関数に分割 |
| 秘密の依存（隠れた依存性） | テスト困難・副作用が見えない | 依存性を明示的に引数または DI で渡す |
| ビッグバン移行 | リスクが高い・部分的ロールバックが困難 | 段階的移行、Strangler Fig パターン |

## 次のステップ

`design-flow` 経由で呼ばれた場合、制御は自動的にオーケストレーターに戻る。

スタンドアロンで起動された場合:

```
アーキテクチャ設計完了。次のステップ:
-> /commit-issue で変更をコミット
-> フルワークフローが必要な場合は /design-flow を使用
```

## 注意事項

- **設計判断がこのスキルの最優先事項** -- 実装の詳細は `code-issue` の責務
- **ビルド検証は不要** -- このスキルは設計ドキュメントを生成し、実行可能コードは作成しない
- Design Brief が渡された場合はそれに基づいて設計。スタンドアロン時は Issue から要件を把握してから設計
- フレームワーク固有の設計（Next.js、Drizzle、shadcn/ui）は専用スキルに委任する
