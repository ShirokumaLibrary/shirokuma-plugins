# リサーチレビュー基準

## 要件合致性

| 基準 | 評価観点 |
|------|---------|
| 技術スタック互換性 | 推奨パターンがプロジェクトの tech-stack（CLAUDE.md / tech-stack.md）と互換か |
| 既存パターン整合性 | 推奨事項がプロジェクト内の既存実装パターンと矛盾しないか |
| 依存関係互換性 | 推奨ライブラリ/バージョンが既存依存関係と競合しないか |
| アーキテクチャ適合性 | 推奨アプローチがプロジェクトのアーキテクチャ方針に沿っているか |

## 調査品質

| 基準 | 評価観点 |
|------|---------|
| ソースの多様性 | 複数の独立したソースから裏付けがあるか（最低 1 つの公式ドキュメント） |
| バージョン整合性 | 参照ドキュメントのバージョンがプロジェクトの使用バージョンと一致するか |
| ソース帰属 | すべての推奨事項にソース URL が付与されているか |
| 最新性 | 古い API やパターンを参照していないか |

## 実装可能性

| 基準 | 評価観点 |
|------|---------|
| 具体性 | 推奨事項がコード例付きで具体的か（抽象的な助言ではないか） |
| 段階的導入 | 大きな変更の場合、段階的な導入パスが示されているか |
| リスク識別 | 潜在的なリスクや注意事項が明示されているか |

## 不合致判定

| 合致度 | 条件 | アクション |
|--------|------|----------|
| 完全合致 | 推奨事項が既存パターンと整合 | そのまま採用 |
| 部分合致 | 一部の推奨事項のみ適用可能 | 適用可能な部分を抽出し、不適用部分の理由を記載 |
| 不合致だが有用 | ベストプラクティスだが現行アーキテクチャと競合 | 取り込み提案を作成（影響範囲・導入パス・リスクを記載） |
| 不合致かつ不適用 | プロジェクトの技術スタックや要件と根本的に合わない | 不採用理由を記載、代替案を探索 |
