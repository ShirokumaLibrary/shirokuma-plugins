# 要件レビューロール

> **注**: このロールは `analyze-issue` に移行しました。現行の実装は `analyze-issue/roles/requirements.md` を参照してください。
> `review-issue requirements` を呼び出した場合、`review-issue` SKILL.md の後方互換スタブが `analyze-issue` に自動委任します。

## 責務

Issue 本文（目的・概要・再現手順・期待動作・成果物・検討事項）の品質レビュー:
- 完全性（必須セクションの有無と内容の充実度）
- 明確性（曖昧な表現・解釈の余地がないか）
- 実行可能性（計画・実装の入力として十分な情報が揃っているか）
- 整合性（セクション間の矛盾や不一致がないか）

## `plan` ロールとの使い分け

| 観点 | `analyze-issue` requirements ロール | `analyze-issue` plan ロール |
|------|-----------------------------------|--------------------------|
| レビュー対象 | Issue 本文（要件・仕様）そのもの | 計画 Issue の計画内容 |
| タイミング | `prepare-flow` 実行前の任意タイミング | 計画策定後のセカンドオピニオン |
| 目的 | 要件・仕様の品質ゲート | 計画の実装妥当性ゲート |
| 起動方法 | `analyze-issue requirements #N` | `analyze-issue plan #N` |

## 必要な知識

以下のファイルをコンテキストとして読み込む:
- プロジェクトの `CLAUDE.md` - プロジェクト概要と規約
- `.claude/rules/` - プロジェクト固有のルール（自動読み込み）

## requirements ロール固有ワークフロー

```
1. ロール選択: "requirements review" or "要件レビュー"
2. Issue 本文取得: shirokuma-docs issue context {number}（→ .shirokuma/github/{org}/{repo}/issues/{number}/body.md を Read ツールで読み込む）
3. Lint 実行: スキップ（対象がコードファイルではないため）
4. Issue 本文分析: 各セクションをレビュー観点で分析
5. レポート生成: テンプレート形式
6. レポート保存: Issue コメント
```

## レビューチェックリスト

### 目的セクションの完全性
- [ ] `## 目的` セクションが存在する
- [ ] 「誰が」（ロール/ユーザー種別）が具体的に記載されている
- [ ] 「何を」（達成したい機能/変更）が明確に記載されている
- [ ] 「なぜ」（理由/動機）が省略されていない
- [ ] 目的が具体的で測定可能（抽象的な表現のみでない）

### 概要セクションの明確性
- [ ] `## 概要` セクションが存在する
- [ ] 変更内容が具体的に説明されている
- [ ] 実装範囲（スコープ）が明確に定義されている
- [ ] スコープ外（対応しないこと）が明記されている（必要な場合）
- [ ] 用語が一貫して使用されている

### 再現手順・期待動作の記述（バグ・不具合の場合）
- [ ] `## 再現手順` が存在する（バグ Issue の場合）
- [ ] 再現手順が番号付きで具体的に記載されている
- [ ] 実際の動作（Actual）と期待する動作（Expected）が明確に分けて記載されている
- [ ] 環境情報（OS、ブラウザ、バージョン等）が記載されている（必要な場合）

### 成果物の具体性
- [ ] `## 成果物` セクションが存在する（Feature/Chore の場合）
- [ ] 各成果物が具体的かつ検証可能な形で記載されている
- [ ] 成果物の完了条件が明確（「〇〇できること」等の形式）
- [ ] 成果物の数・範囲が適切（多すぎず少なすぎず）

### 検討事項の充実度
- [ ] `## 検討事項` セクションが存在する（M 以上の Issue の場合）
- [ ] 技術的な制約・依存関係が記載されている
- [ ] 代替案・除外した選択肢が記載されている（該当する場合）
- [ ] リスクや注意点が明記されている

### 計画・実装の入力として十分か
- [ ] Issue 本文だけで要件の全体像を理解できる
- [ ] 実装者が判断に迷う曖昧な箇所がない
- [ ] 外部コンテキスト（別の Issue、PR、会話等）に依存しすぎていない
- [ ] 優先度・緊急度が適切に設定されている

### プロジェクト要件整合性

ADR 参照のトリガー条件（以下のいずれかに該当する場合に実施する）:
- Issue タイトルまたは本文に「ADR」「アーキテクチャ」「設計決定」「技術選定」「architecture」「design decision」等のキーワードを含む
- ラベルに `area:plugin`・`area:workflow`・`area:cli` 等の構造変更が想定される area ラベルを含み、かつ本文に上記キーワードが共存する

ADR 絞り込みロジックは [docs/adr-filter-logic.md](../docs/adr-filter-logic.md) を参照。

チェック項目:

1. - [ ] 関連する Accepted 状態の ADR が存在し、本 Issue の方針と矛盾していないか（`discussion adr list` + 上位 5 件の詳細確認）
2. - [ ] 過去の命名規則・制約 Issue（「命名規則」「規約」「naming」「convention」等のキーワードで `issue search` 検索、Closed 含む）に準拠しているか
3. - [ ] 本 Issue で提案する技術選定・アーキテクチャ方針が、既存 ADR で Deprecated/Superseded とされた選択肢を再採用していないか
4. - [ ] スコープ・責務範囲が、既存 ADR で明示された境界（例: CLI ↔ スキルの責務分離 ADR）に沿っているか
5. - [ ] 提案する変更が、他の進行中 Issue（In Progress / Review ステータス）と競合・矛盾していないか（`issue list --status "In progress"` で確認）
6. - [ ] ADR の整合性チェックで「矛盾なし」と判定した根拠（参照した ADR 番号）をレポートに明記しているか
7. - [ ] 新規スキル・ルール追加の場合、既存スキルとのトリガーキーワード重複がないか（`skill-authoring.md` 参照）
8. - [ ] 本 Issue がエピックの場合、サブ Issue 間の整合性（矛盾する成果物・競合するスコープ）がないか

**構造化出力（整合性チェック実施時のみ）:**
- `**プロジェクト要件整合性:**` PASS / NEEDS_REVISION
- `**参照 ADR:**` ADR-{NNN}, ADR-{NNN}, ...（参照した ADR 番号リスト）

## 検出すべきアンチパターン

### 目的の欠落
- [ ] `## 目的` セクションが存在しない
- [ ] 目的が「〇〇を修正する」のみで動機が不明
- [ ] 「誰が困っているか」が記載されていない

### 曖昧な概要
- [ ] 「改善する」「最適化する」のみで具体的な変更内容なし
- [ ] 実装範囲が不明確（何をスコープとするか不明）
- [ ] 専門用語の定義がなく解釈の余地がある

### 成果物の不明確さ
- [ ] `## 成果物` セクションが空、または「動作すること」のみ
- [ ] 成果物が検証不可能な表現（「使いやすくなること」等）
- [ ] 複数の成果物が混在して区別できない

### 実装困難な要件
- [ ] 要件同士に矛盾がある
- [ ] 技術的に実現不可能な要件が含まれている
- [ ] 依存関係が未解決のまま記載されている

## 設計要否判定

要件レビューの過程で、課題が設計フェーズを必要とするかを判定し、構造化出力として `**設計要否:** NEEDED / NOT_NEEDED` を追加する。

### 判定条件

| 観点 | 判定 |
|------|------|
| 要件に既存パターンで対応できない新規性がある | NEEDED |
| UI / 画面の新規作成・大幅な変更 | NEEDED |
| データモデル・スキーマの追加・変更 | NEEDED |
| 外部 API・連携の新規追加 | NEEDED |
| `area:frontend` / `area:ui` ラベルがある | NEEDED |
| `area:database` ラベルがある | NEEDED |
| 本文キーワード: `UI`, `画面`, `スキーマ`, `データモデル`, `screen`, `schema`, `data model` | NEEDED（キーワードが新規作成・変更文脈で使われている場合） |
| 既存コードのリファクタリング・軽微な修正 | NOT_NEEDED |
| 設定変更・ドキュメント追加・バグ修正 | NOT_NEEDED |
| 上記 NEEDED 条件に該当しない | NOT_NEEDED |

### 判定優先順序

NEEDED 条件に 1 つでも該当すれば `NEEDED` と判定する（AND ではなく OR）。キーワードベースの判定はラベルベースの判定より弱い補助判定として扱い、キーワードが「既存機能の修正」文脈で使われている場合は NEEDED とは判定しない。

## レポート形式

`templates/report.md` のテンプレートを使用:

1. **概要**: Issue 本文の品質サマリー
2. **重大な問題**: 必須セクションの欠落、致命的な曖昧さ、実装困難な要件
3. **改善提案**: 記述の充実化、具体性の向上、整合性の修正
4. **良い実践**: 適切な記述パターン
5. **推奨事項**: 優先度順のアクション項目
