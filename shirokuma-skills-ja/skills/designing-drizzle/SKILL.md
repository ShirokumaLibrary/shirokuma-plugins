# Drizzle データモデル設計

パターン選択に基づいた Drizzle ORM のデータモデル設計。設計判断とトレードオフ分析に集中し、実装は `coding-on-issue` に委任する。

> **データモデル設計はこのスキルの責務。** `coding-on-issue` はここで決定されたスキーマに基づいて実装を担当する。

## ワークフロー

### 0. 技術スタック確認

**最初に**、プロジェクトの `CLAUDE.md` を読んで確認:
- Drizzle ORM バージョン
- データベース（PostgreSQL / MySQL / SQLite）
- マイグレーションツール（drizzle-kit）
- 認証（Better Auth、NextAuth — セッション/ユーザーテーブルの制約）
- 既存スキーマファイルの構成（`src/db/schema/` 等）

`.claude/rules/` 内の `tech-stack.md` も確認する。

### 1. 設計コンテキスト確認

`designing-on-issue` から委任された場合、Design Brief と要件が渡される。そのまま使用する。

スタンドアロンで起動された場合、Issue 本文と計画セクションから要件を把握する。

### 2. データモデル分析

Issue に関連する各設計観点について、決定フレームワークを適用する:

#### 設計観点

| 観点 | 対処するタイミング | パターン参照 |
|------|------------------|-------------|
| テーブル設計 | 新規エンティティ、正規化レベル | [patterns/data-model-patterns.md](patterns/data-model-patterns.md) - テーブル設計 |
| リレーション設計 | エンティティ間の関連、結合戦略 | [patterns/data-model-patterns.md](patterns/data-model-patterns.md) - リレーション |
| インデックス戦略 | クエリパフォーマンス、検索条件 | [patterns/data-model-patterns.md](patterns/data-model-patterns.md) - インデックス |
| ソフトデリート | 論理削除が必要なエンティティ | [patterns/data-model-patterns.md](patterns/data-model-patterns.md) - ソフトデリート |
| マイグレーション | スキーマ変更、データ移行 | [patterns/data-model-patterns.md](patterns/data-model-patterns.md) - マイグレーション |

#### 決定フレームワーク

各観点について評価する:

1. **要件**: 機能が必要とするデータ構造は?
2. **制約**: DB エンジン、既存スキーマ、パフォーマンス要件
3. **選択肢**: 実行可能なパターンを列挙（data-model-patterns.md 参照）
4. **トレードオフ**: 決定マトリクスでオプションを比較
5. **決定**: 根拠を添えてパターンを選択

### 3. 設計出力

構造化されたドキュメントとしてデータモデル設計を作成する:

```markdown
## データモデル設計

### エンティティ一覧
| エンティティ | 説明 | 主要カラム |
|------------|------|-----------|
| {name} | {purpose} | {key columns} |

### テーブル定義
{各テーブルの Drizzle スキーマ定義（型、制約、デフォルト値）}

### リレーション
{エンティティ間のリレーション定義と結合戦略}

### インデックス
{パフォーマンスに必要なインデックス定義と根拠}

### マイグレーション戦略
{スキーマ変更の適用順序とデータ移行手順}

### 主要決定事項
| 決定 | 選択 | 根拠 |
|------|------|------|
| {トピック} | {パターン} | {理由} |
```

### 4. レビューチェックリスト

- [ ] テーブル名が snake_case で複数形
- [ ] 主キーが適切（UUID vs 連番）
- [ ] 外部キー制約と ON DELETE 動作が明示的
- [ ] インデックスがクエリパターンに合致
- [ ] ソフトデリートが必要なテーブルに `deletedAt` カラムあり
- [ ] マイグレーションが後方互換（必要な場合）
- [ ] 既存スキーマパターンと整合

## アンチパターン

| パターン | 問題 | 代替案 |
|---------|------|--------|
| 過度な正規化 | JOIN 過多でパフォーマンス低下 | 読み取り頻度の高いデータは非正規化を検討 |
| インデックスなしの外部キー | 結合クエリが遅い | 外部キーカラムにインデックスを追加 |
| 全テーブルにソフトデリート | クエリ複雑化、WHERE 条件漏れ | 必要なテーブルのみに適用 |
| 文字列型のステータスカラム | 型安全性なし、タイポリスク | Drizzle の enum 型を使用 |
| マイグレーションでのデータ変更 | ロールバック困難 | スキーマ変更とデータ移行を分離 |

## 次のステップ

スタンドアロンで起動された場合:

-> フルワークフローが必要な場合は /designing-on-issue を使用

## 注意事項

- **設計判断がこのスキルの最優先事項** -- 実装の詳細は `coding-on-issue` の責務
- **ビルド検証は不要** -- このスキルは設計ドキュメントを生成し、実行可能コードは作成しない
- Design Brief が渡された場合はそれに基づいて設計。スタンドアロン時は Issue から要件を把握してから設計
- DB エンジン固有の制約は必ず確認（PostgreSQL の配列型、SQLite の ALTER TABLE 制限等）
