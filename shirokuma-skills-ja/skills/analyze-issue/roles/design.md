# 設計レビューロール

## 責務

設計成果物（Design Brief、Aesthetic Direction、UI 実装）の品質レビュー:
- Design Brief の品質（目的、コンテキスト、差別化要素の明確さ）
- Aesthetic Direction の妥当性（トーン選択、デザイントークン定義）
- 要件との整合性
- 技術的実現可能性（shadcn/ui, Tailwind CSS v4 等のフレームワーク制約）
- UI 実装品質（コンポーネント構成、レスポンシブ対応）

## `design-flow` との使い分け

| 観点 | `design-flow` 内蔵チェック | `analyze-issue` design ロール |
|------|----------------------------------|-----------------------------------|
| タイミング | 設計フェーズ中の反復チェック | 設計完了後のゲートレビュー |
| 目的 | 設計の方向性確認 | 独立したセカンドオピニオン |
| 起動方法 | `design-flow` のループ内 | `/analyze-issue design #N` |

## 必要な知識

以下のファイルをコンテキストとして読み込む:
- プロジェクトの `CLAUDE.md` - プロジェクト概要と規約
- `.claude/rules/` - プロジェクト固有のルール（自動読み込み）

## design ロール固有ワークフロー

```
1. ロール選択: "design review" or デザイン関連 Issue
2. Issue 本文取得: shirokuma-docs issue context {number}（→ .shirokuma/github/{org}/{repo}/issues/{number}/body.md を Read ツールで読み込む）
3. Lint 実行: スキップ（対象がコードファイルではないため）
4. 設計分析: Design Brief, Aesthetic Direction, UI 実装結果をレビュー
5. レポート生成: テンプレート形式
6. レポート保存: Issue コメント
```

## レビューチェックリスト

### Design Brief 品質
- [ ] Design Brief が存在する
- [ ] Purpose が明確（解決する問題が具体的に記載されている）
- [ ] Context が網羅的（技術的制約、既存デザインシステムとの整合性が記載されている）
- [ ] Differentiation が具体的（抽象的でない独自性が定義されている）
- [ ] ターゲットユーザーが明確に定義されている

### Aesthetic Direction 妥当性
- [ ] Tone が明確に選択されている（例: professional, playful, minimal）
- [ ] デザイントークンが定義されている（色、間隔、タイポグラフィ）
- [ ] shadcn/ui テーマとの整合性がある
- [ ] プロジェクトの既存 UI との一貫性が保たれている
- [ ] ボーダー、シャドウ等の装飾要素が定義されている

### 要件整合性
- [ ] Issue の要件が設計で全てカバーされている
- [ ] 設計が要件のスコープを超えていない（スコープクリープ）
- [ ] ユーザーストーリー/ペルソナとの整合がある

### 技術的実現可能性
- [ ] 使用フレームワーク（shadcn/ui, Tailwind CSS v4）の制約に準拠している
- [ ] Radix UI ハイドレーションパターンへの考慮がある
- [ ] レスポンシブデザインの考慮がある
- [ ] アクセシビリティ（a11y）基準への準拠がある

### UI 実装品質（実装後レビュー時）
- [ ] コンポーネント分割が適切である
- [ ] デザイントークンが一貫して使用されている
- [ ] ダークモード/ライトモードの考慮がある（該当する場合）
- [ ] アニメーション/トランジションが適切である（パフォーマンスへの影響を含む）

## 検出すべきアンチパターン

### Design Brief 不在
- [ ] Design Brief が存在しない
- [ ] Brief が空または不十分（見出しのみで内容なし）

### デザイントークン未定義
- [ ] 色やスペーシングがハードコードされている
- [ ] 既存のデザインシステムとの不整合がある
- [ ] Tailwind CSS v4 のユーティリティではなく直接値が使用されている

### 要件との乖離
- [ ] 設計が要件を満たしていない
- [ ] 過剰な装飾（パフォーマンスリスク）がある
- [ ] スコープ外の機能が設計に含まれている

### アクセシビリティ違反
- [ ] コントラスト不足がある
- [ ] キーボードナビゲーション未対応である
- [ ] aria ラベルが欠如している
- [ ] フォーカス管理が不適切である

## レポート形式

`templates/report.md` のテンプレートを使用:

1. **概要**: 設計品質の全体サマリー
2. **重大な問題**: Design Brief 不在、要件未カバー、アクセシビリティ違反
3. **改善提案**: デザイントークン追加、レスポンシブ改善等
4. **良い実践**: 適切な設計パターン
5. **推奨事項**: 優先度順のアクション項目
