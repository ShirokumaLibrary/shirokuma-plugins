# リサーチ完了後フロー

リサーチ完了後の条件分岐ロジックを定義する。呼び出しコンテキストによりフローが異なる。

## 呼び出しコンテキスト別フロー

### チェーン経由（`working-on-issue`）

`working-on-issue` がフロー全体を制御する。リサーチスキルは調査結果を返すのみ。

```
researching-best-practices → review-issue(research, Skill) → Discussion 自動保存 → チェーン完了
```

- Discussion 保存: **必須**（`working-on-issue` の `researching-reference.md` が実行）
- 次アクションの判定: 不要（チェーンが完了で終了）

### スタンドアロン

ユーザーが直接 `/researching-best-practices` で起動した場合。

```
researching-best-practices → [Discussion 保存（推奨）] → 次アクション提案
```

- Discussion 保存: **推奨**（任意）。軽微な確認（1 分で完了する調査）は保存不要
- 次アクション: 調査結果に基づき条件分岐テーブルで判定し提案

## 条件分岐テーブル

リサーチ完了後、調査結果の性質に応じて次のアクションを判定する。条件は優先順位順に記載されており、複数の条件に該当する場合は上位（先頭）のアクションを優先する。

| 条件 | 判定基準 | 次のアクション | 具体例 |
|------|---------|---------------|--------|
| 実装タスクが特定された | 調査結果に「X を実装すべき」「Y を追加すべき」等の具体的な作業項目がある | Issue 作成（Backlog） | 「CSP ヘッダーを追加すべき」→ Issue 化 |
| アーキテクチャ判断が必要 | 複数の選択肢があり、トレードオフの比較が含まれる | ADR Discussion 作成 | 「認証方式: JWT vs セッション」→ ADR |
| 再利用可能なパターンが確認された | 特定の実装パターンが検証済みで他の箇所にも適用可能 | Knowledge Discussion 作成 | 「Drizzle のソフトデリートパターン」→ Knowledge |
| 既存ルールの改善が示唆された | 調査結果が既存ルール・スキルの不備や改善点を含む | Evolution シグナル記録 | 「現行のバリデーションルールが不十分」→ Evolution Issue |
| パターンが 3+ 件蓄積 | Knowledge Discussion に同種のパターンが 3 件以上存在 | Rule 化提案（`managing-rules`） | 3 つの i18n パターン → ルール抽出 |
| 情報提供のみ | 上記のいずれにも該当しない純粋な情報共有 | Discussion のまま（追加アクション不要） | 「ライブラリ X のバージョン互換性」→ そのまま |

## 判定基準の詳細

### 「実装タスクが特定された」の判定

以下のいずれかに該当する場合:
- 調査結果に具体的なコード変更が含まれる（ファイル名・関数名レベル）
- 「〜すべき」「〜が必要」等の行動指示がある
- セキュリティ脆弱性やバグの修正が必要

### 「アーキテクチャ判断が必要」の判定

以下のいずれかに該当する場合:
- 2 つ以上の実装方式の比較がある
- 判断の影響範囲が複数ファイル・モジュールにまたがる
- 可逆性が低い（後から変更するコストが高い）

### 「再利用可能なパターン」の判定

以下のすべてに該当する場合:
- 特定のユースケースに限定されない汎用的なパターン
- 実際のコードで検証済み（理論のみでない）
- 他の開発者（AI エージェント含む）が参照する価値がある

## スタンドアロン完了時の提案テンプレート

```markdown
## 次のステップ

調査結果に基づき、以下のアクションを推奨します:

- [該当するアクションを条件分岐テーブルから選択して提案]
```

複数のアクションが該当する場合は、優先度の高い順に提案する。
