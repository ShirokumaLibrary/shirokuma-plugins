# デザイン評価

実装完了後にユーザーによる視覚評価を実施するスキル。`design-flow` の Phase 4 として呼び出される。

## コンテキスト

呼び出し元 `design-flow` から以下のコンテキストが渡される:

- 変更されたファイルパス一覧
- 設計タイプ（UI 設計かどうか）

## スキップ条件

視覚要素を持たない設計タイプ（データモデル設計等）の場合、呼び出し元がこのスキルをスキップして Phase 5 に直行する。

## ワークフロー

### ステップ 1: dev サーバー確認

```bash
# dev サーバーが起動しているか確認
lsof -i :3000 2>/dev/null || echo "dev server not running"
```

dev サーバーが起動していない場合は起動を提案する。

### ステップ 2: ユーザーレビュー

`AskUserQuestion` で以下を提示:

- 変更ファイルパス一覧
- 確認用 URL（dev サーバーが稼働している場合: `http://localhost:3000`）
- レビューチェックリスト:
  - [ ] タイポグラフィが特徴的
  - [ ] カラーパレットが統一的
  - [ ] モーション/アニメーションの印象
  - [ ] レイアウトの視覚的面白さ
  - [ ] 全体的な印象

選択肢を提示:
1. **承認** → 視覚評価完了、呼び出し元に返す
2. **修正依頼** → 修正内容を受け取り呼び出し元に戻す（Phase 3 でデザインスキルに再委任）
3. **方向性変更** → 方向性変更を呼び出し元に通知する（Phase 2 に戻る）

### 参考: 安全上限

視覚評価ループは **最大 3 イテレーション**。

呼び出し元 `design-flow` がイテレーション数を管理し、上限到達時はこのスキルを呼び出さずに Phase 5 に直行する。上限到達時は現状で進行し、フォローアップ Issue での改善を提案する。

## 出力

ユーザーの選択に応じて以下を呼び出し元（`design-flow`）に返す:

| 選択 | 返す内容 |
|------|---------|
| 承認 | `APPROVED` |
| 修正依頼 | `NEEDS_REVISION: {修正内容}` |
| 方向性変更 | `DIRECTION_CHANGE` |

## ツール使用

| ツール | タイミング |
|--------|-----------|
| Bash | dev サーバー確認（ステップ 1） |
| AskUserQuestion | 視覚評価（ステップ 2） |

## 注意事項

- このスキルは `AskUserQuestion` を使用するため、Skill ツール（メインコンテキスト）で呼び出す必要がある（Agent 委任は不可）
- イテレーション数の管理は呼び出し元 `design-flow` が行う
