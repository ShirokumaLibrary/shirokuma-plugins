# アーキテクチャパターン

Next.js アーキテクチャ設計判断のためのパターン比較テーブル。

## ルーティング設計

### ルート構成

| パターン | 使用場面 | トレードオフ |
|---------|---------|------------|
| フラットルート (`app/page/page.tsx`) | シンプルなアプリ、少数ページ | 簡単だがスケールしない |
| ルートグループ (`app/(group)/page/`) | 共有レイアウト、認証境界 | ディレクトリ増だが明確な境界 |
| パラレルルート (`@slot`) | ダッシュボードパネル、モーダル | 複雑だが独立したローディングが可能 |
| インターセプティングルート (`(.)path`) | 共有可能 URL 付きモーダルオーバーレイ | 強力だがデバッグ困難 |

### レイアウト境界

| パターン | 使用場面 | 例 |
|---------|---------|-----|
| ルートレイアウト | グローバルシェル（ナビ、フッター） | `app/layout.tsx` |
| ルートグループレイアウト | 認証ゲート付きセクション | `app/(dashboard)/layout.tsx` |
| ネストレイアウト | セクション固有の装飾 | `app/(dashboard)/settings/layout.tsx` |
| レイアウトなし（ページのみ） | 独立ページ（ログイン、エラー） | `app/login/page.tsx` |

### 判定: ルートグループ vs フラット

| 基準 | ルートグループ | フラット |
|------|-------------|--------|
| 共有レイアウトが必要 | 推奨 | レイアウト重複 |
| 認証境界 | 推奨 | ミドルウェアのみ |
| 関連ページ 5+ | 推奨 | 許容可 |
| 独立ページ 1-3 | オーバーヘッド | 推奨 |

## コンポーネント階層

### Server vs Client コンポーネント境界

| パターン | 使用場面 | 例 |
|---------|---------|-----|
| Server Component（デフォルト） | 静的コンテンツ、データ取得、非インタラクティブ | ページ、レイアウト、データ表示 |
| Client Component (`"use client"`) | イベントハンドラ、useState、useEffect、ブラウザ API | フォーム、インタラクティブウィジェット |
| コンポジション（Server が Client を包む） | 静的 + インタラクティブの混在 | Server がデータ取得し Client フォームに渡す |
| Client-only サブツリー | 高度にインタラクティブなセクション | ドラッグ&ドロップ付きダッシュボード |

### コンポーネント境界ルール

```
Page (Server)
  -> レイアウトセクション (Server)
    -> データ表示 (Server)
    -> インタラクティブウィジェット (Client)
      -> Client の子要素 (Client - 境界は継承)
```

**基本原則**: `"use client"` は可能な限り深い階層に配置。データ取得コンポーネントは Server Component に保つ。

### コンポジションパターン

| パターン | 使用場面 | 例 |
|---------|---------|-----|
| Container/Presentational | データ取得と表示の分離 | `UserContainer` (Server) -> `UserCard` (Client) |
| スロットパターン | 柔軟なレイアウト構成 | `<Layout sidebar={<Sidebar />}>` |
| children 経由のレンダープロップス | Server Component が Client にコンテンツを渡す | `<Modal>{serverContent}</Modal>` |
| 並列データ取得 | 独立したデータソース | `Promise.all([getUser(), getPosts()])` |

## データレイヤー

### Server Actions vs API Routes

| 基準 | Server Actions | API Routes |
|------|---------------|------------|
| 同一オリジンのフォーム送信 | 推奨 | オーバーヘッド |
| 外部 API 連携 | 可能だが不自然 | 推奨 |
| Webhook エンドポイント | 非対応 | 推奨 |
| プログレッシブエンハンスメント | 組み込み | 手動対応 |
| ファイルアップロード | 対応（FormData） | 対応 |
| サードパーティクライアントアクセス | 非対応 | 推奨 |

### Server Action パターン

| パターン | 使用場面 | 構成 |
|---------|---------|------|
| ファイル単位の単一アクション | シンプルな CRUD | `lib/actions/create-user.ts` |
| グループ化アクション | 関連する操作 | `lib/actions/user-actions.ts` |
| 再検証付きアクション | 変更後 | `revalidatePath()` / `revalidateTag()` |
| リダイレクト付きアクション | 作成/削除後 | `redirect('/path')` |

### Server Actions セキュリティチェックリスト

1. **認証チェック**: 先頭でセッションを検証
2. **CSRF 防御**: フレームワークレベル（Server Actions に組み込み）
3. **入力バリデーション**: Zod スキーマでバリデーション
4. **認可**: 所有権/権限を検証
5. **レート制限**: 破壊的操作に対して実施

## ミドルウェア

### ミドルウェアの責務

| 責務 | 適切か | 理由 |
|------|--------|------|
| 認証リダイレクト（未認証 -> ログイン） | はい | ページレンダリング前に実行 |
| i18n ロケール検出 | はい | 全ルートで実行が必要 |
| セキュリティヘッダー | はい | グローバルな関心事 |
| フィーチャーフラグ（ヘッダーベース） | はい | 低オーバーヘッド |
| 重いデータ取得 | いいえ | 全ルートをブロック |
| ページ単位の認可 | いいえ | レイアウト/ページレベルのチェックを使用 |
| 複雑なビジネスロジック | いいえ | テスト・デバッグが困難 |

### ミドルウェア実行順序

```
リクエスト
  -> 1. セキュリティヘッダー
  -> 2. i18n ロケール検出
  -> 3. 認証チェック（未認証ならリダイレクト）
  -> 4. フィーチャーフラグ
  -> ページレンダリング
```

### Matcher 設定

| パターン | 使用場面 |
|---------|---------|
| `matcher: ['/dashboard/:path*']` | 特定セクションを保護 |
| `matcher: ['/((?!api\|_next\|static).*)']` | 静的/API 以外を保護 |
| ミドルウェア本体内の条件分岐 | 複雑なパスベースロジック |

## データフロー

### 状態管理の選択

| アプローチ | 使用場面 | トレードオフ |
|-----------|---------|------------|
| Server Component props | 静的またはサーバー取得データ | 最もシンプル、クライアント JS なし |
| URL search params | フィルタ/ソート可能なリスト | 共有可能だが型が限定的 |
| React Context | 共有クライアント状態（テーマ、認証） | 変更時に再レンダリング |
| Zustand/Jotai | 複雑なクライアント状態 | 追加依存関係 |
| Server Actions + revalidation | 変更 -> 再取得サイクル | フレームワークネイティブ |

### キャッシュ戦略

| レイヤー | メカニズム | 使用場面 |
|---------|-----------|---------|
| リクエスト重複排除 | Server Components での `fetch` | 複数コンポーネントで同じリクエスト |
| データキャッシュ | `revalidate` オプション付き `fetch` | API レスポンス |
| フルルートキャッシュ | 静的/動的レンダリング | ページ全体 |
| ルーターキャッシュ | クライアントサイドナビゲーション | 以前訪問したルート |

### 再検証パターン

| パターン | 使用場面 |
|---------|---------|
| `revalidatePath('/path')` | 特定ページに影響する変更後 |
| `revalidateTag('tag')` | タグ付き fetch に影響する変更後 |
| `router.refresh()` | クライアントサイド強制リフレッシュ |
| 時間ベース (`revalidate: 60`) | 半静的データの stale-while-revalidate |
