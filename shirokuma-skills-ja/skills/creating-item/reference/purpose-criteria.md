# 目的明確性の判定基準

JTBD（Jobs-to-be-Done）フレームワークに基づき、ユーザーの発話が「手段」なのか「目的」なのかを判定する。

## 手段 vs 目的の違い

| | 手段（What to do） | 目的（Why / Outcome） |
|--|------------------|--------------------|
| 焦点 | 実装・操作の内容 | 受益者が得られる価値 |
| 問い | 「何をするか」 | 「誰が・何を・なぜできるか」 |
| 例 | 「show コマンドを集約したい」 | 「ユーザーがシンプルなコマンド体系で操作できるようにする。現在分散しており、混乱が生じているため。」 |

## 手段パターンの検出

以下のパターンに一致する発話は「手段」と判定する。

### 動詞パターン（JA）

| パターン | 例 |
|---------|-----|
| 〜を修正したい | 「バグを修正したい」 |
| 〜を追加したい | 「オプションを追加したい」 |
| 〜を集約したい / 〜を統合したい | 「コマンドを集約したい」 |
| 〜を検討したい | 「アーキテクチャを検討したい」 |
| 〜を実装したい | 「フィーチャーを実装したい」 |
| 〜を変更したい | 「インターフェースを変更したい」 |
| 〜をリファクタリングしたい | 「コードをリファクタリングしたい」 |
| 〜を整理したい | 「ファイル構造を整理したい」 |
| 〜を移行したい | 「ライブラリを移行したい」 |
| 〜を更新したい | 「依存パッケージを更新したい」 |
| 〜を削除したい | 「不要なコードを削除したい」 |

### 文脈パターン

- 「〜してほしい」（依頼形）
- 「〜する必要がある」（義務形）
- 技術的な対象（ファイル名、コマンド名、API 名）のみを述べている場合

## 判定フロー

```
ユーザーの発話を受け取る
    │
    ▼
手段パターンに一致するか？
    │
    ├─ YES → 手段を検出
    │            │
    │            ▼
    │        目的を推定する（以下の情報から）
    │            - 会話の前後のコンテキスト
    │            - 変更対象のコードや仕組みの役割
    │            - 問題の根本原因（"なぜその手段が必要か"）
    │            │
    │            ▼
    │        AskUserQuestion で確認する
    │            - 推定した目的を 2〜3 候補提示
    │            - 「その他（具体的に）」を必ず含める
    │
    └─ NO  → 「誰が・何を・なぜ」が含まれるか確認
                 │
                 ├─ 全て含まれる → そのまま Issue 作成へ
                 └─ 不足あり    → 候補を提示して確認
```

## 目的の要素

優れた目的は以下の 3 要素を含む（JTBD の "Job Statement" に準拠）:

| 要素 | 説明 | 例 |
|------|------|-----|
| 受益者 | 誰が恩恵を受けるか | CLI ユーザー、開発者、チーム |
| 期待結果 | 何ができるようになるか | 〜できるようにする |
| 動機 | なぜそれが必要か | 〜のため、〜だから |

## 目的候補の推定方法

手段パターンを検出したら、以下の手順で目的候補を推定する:

1. **変更対象の「役割」を把握する**: コマンド、スキル、ファイルが誰のために何をするものか
2. **「なぜその手段が必要か」を問い直す**: 現状の何が問題か、何が不便か
3. **受益者を特定する**: エンドユーザー、開発者、チームのどれか
4. **複数の仮説を立てる**: ユーザーが意図している可能性が高い目的を 2〜3 パターン作成

## 例

### 例 1: コマンド集約

```
ユーザー: 「CLI の show が個別のと纏まってるのとで分散してるので、纏まってる方に集約する検討をしたい」

→ 手段パターン検出: 「集約する検討をしたい」

→ 推定した目的候補:
A) 「ユーザーがシンプルなコマンド体系でコンテンツを取得できるようにする。
    現在 show と個別コマンドで機能が分散しており、どちらを使えば良いか分かりにくいため。」
B) 「開発者がコンテキストサイズを削減しながら実装を保守できるようにする。
    重複実装により保守コストが高く、ドキュメントも分散しているため。」
C) その他（具体的に教えてください）
```

### 例 2: オプション追加

```
ユーザー: 「deps コマンドに --format オプション追加して」

→ 手段パターン検出: 「追加して」（依頼形）

→ 推定した目的候補:
A) 「CLI ユーザーが出力形式を用途に応じて選べるようにする。
    現在 SVG のみで、CI やドキュメントへの埋め込みに不便なため。」
B) 「CLI ユーザーが JSON 形式で依存情報を取得できるようにする。
    他ツールとのパイプライン連携のため。」
C) その他（具体的に教えてください）
```

### 例 3: 手段パターン非検出（目的が明確）

```
ユーザー: 「ユーザーが設定ファイルなしで CLI を使い始めれるようにしたい。
           初期設定の手間が導入障壁になっているため」

→ 手段パターン非検出（「何を・なぜ」が明確）
→ そのまま Issue 作成へ
```

## 過検出の防止

手段パターンに一致しても、以下の場合は確認をスキップする（文脈から目的が十分に明確）:

| ケース | 判断 |
|--------|------|
| 既に会話で目的が述べられている | 確認不要 |
| XS サイズ（タイポ修正、1行変更）| 確認不要（1文の目的で十分） |
| ユーザーが「Issue にして」と指定し発話の前に目的説明がある | 確認不要 |
