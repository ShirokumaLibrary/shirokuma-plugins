# GitHub ライティングスタイル

GitHub 書き込み（Issues, PRs, Discussions の本文・コメント）における記述スタイル。

## 散文と箇条書きの使い分け

| 条件 | 形式 |
|------|------|
| 3+ 項目の並列情報 | 箇条書き |
| 手順・順序あり | 番号付きリスト |
| 因果関係・論理展開 | 散文 |
| 単一の説明文 | 散文 |
| 変更の全体像・コンテキスト | 散文（概要段落） |

## 散文的な概要

リスト・テーブルを使う前に、変更・分析・提案の**結論を先に述べる1〜2文の概要段落**を書く。

```
# 良い例（概要 + 詳細）
{何をしたか・何が変わるか} を1〜2文で述べた後、詳細を箇条書きやテーブルで補足する。
```

```
# 悪い例（概要なし）
いきなり箇条書きが始まり、変更全体の意図が読み取れない。
```

**概要段落が有効な場面:**
- Issue・PR の本文冒頭
- レビューコメントで複数の指摘をまとめるとき
- 設計変更・破壊的変更の説明

## 対比による明確化

「Before / After」または「現状 / 改善後」の対比構造は、変更の意図を端的に伝える。

```markdown
**現状**: リスト前の導入文がなく、コンテキストが欠如している。
**改善後**: 導入文を追加し、リストの目的を明示する。
```

複数の対比がある場合はテーブルで表現する。

## フォーマットルール

- **並列構造の統一**: 各項目を同じ品詞・構造で開始する
- **ネスト上限**: 2段まで（3段以上は分割またはテーブルに変換）
- **リスト長**: 1リストあたり 2-7 項目（8+ はグループ分割を検討）
- **導入文**: リスト前に1文のコンテキストを追加する
- **結論ファースト**: 概要段落は「何がどう変わるか」を先に述べる

## 識別子・参照の記法

GitHub の Issue / PR / Discussion 本文・コメントでは `#数字` は自動的に Issue/PR/Discussion へのリンクになる。意図しない自動リンクを防ぐ記法ルール。

| 対象 | 記法 |
|------|------|
| Issue / PR / Discussion の参照 | `#123` |
| 通し番号（問題リスト・識別子） | `P-1`, `問題1`, `G1`（`#` を含めない） |
| ローカル識別子 | 英字プレフィックス + 数字（`D1`, `Q2` 等） |

**アンチパターン**: 問題テーブルで `#1`, `#2` と書くと全て Issue 自動リンクとして解釈される。通し番号に `#` を使わない。

## アンチパターン

| パターン | 問題 | 修正 |
|---------|------|------|
| Wall of Bullets | 全文を箇条書き化、要点が埋没 | 散文概要 + 箇条書きに変換 |
| 並列構造の崩れ | 項目ごとに異なる構文 | 統一した文法構造で書き直す |
| 導入文なしリスト | コンテキスト欠如で意味不明 | リスト前に概要文を1文追加 |
| 単一項目リスト | 箇条書きの意味がない | 散文に変換 |
| 概要なしリスト開始 | 変更の意図・範囲が不明 | 1〜2文の概要段落を冒頭に追加 |

## Mermaid 図

GitHub は Mermaid をネイティブレンダリングする。複雑な構造やフローの説明に図を活用する。

### 使用条件

| 条件 | Mermaid を含める | 図の種類 |
|------|-----------------|---------|
| 3+ タスクに依存関係がある | はい | フローチャート |
| 状態遷移の記述がある | はい | 状態図 |
| 2+ コンポーネント間のやりとりがある | はい | シーケンス図 |
| 単純な直線的タスクリスト | いいえ | — |
| 単一ファイル変更 | いいえ | — |

### スタイル規約

- **ノード数上限**: 1つの図は10ノード以下。超える場合はサブ図に分割
- **ラベル**: 簡潔に（1ノード10文字以内を目安）
- **方向**: フローチャートは `graph TD`（上→下）または `graph LR`（左→右）を内容に応じて選択
- **コードブロック**: ` ```mermaid ` で囲む
- **改行**: ラベル・ノート内の改行は `<br/>` を使う。`\n` は GitHub レンダラーでリテラル表示される
- **背景色**: `rect rgb(...)` や `style fill:...` でノード背景色をカスタマイズしない。明るい背景と白いテキストが重なりダークモードで読めなくなる

### アンチパターン

| パターン | 問題 | 修正 |
|---------|------|------|
| テキストで十分な内容を図にする | 冗長、メンテナンスコスト増 | 条件テーブルに該当する場合のみ図を使う |
| 10ノード超の巨大図 | 視認性低下 | サブ図に分割 |
| 図だけで説明を省略 | コンテキスト欠如 | 図の前後に1文の説明を追加 |

## スコープ

GitHub Issues / PRs / Discussions の本文・コメントに適用。スキル完了レポート（`completion-report-style` ルール）は対象外。Claude 設定ファイルの構造については `coding-claude-config` スキルを参照。
