# コンテキスト効率化分類

コンテキスト消費量に基づいて Agent 委任と Skill 呼び出しのどちらを選ぶかのガイドライン。

## 分類基準

| 判断基準 | ルート | 根拠 |
|---------|-------|------|
| 多数のファイル読み書き（10件以上）または大容量ファイル | Agent ツール | メインコンテキストの肥大化を防ぐ |
| 外部 Web アクセス（WebSearch、WebFetch） | Agent ツール | Web コンテンツが予測不能にコンテキストを膨張させる |
| 200 行超の変更 | Agent ツール | 大きな差分をメインコンテキストに置かない |
| 軽量タスク + プロジェクトルールが必要 | Skill ツール | ルールはメインコンテキストでのみアクセス可能 |
| git/GitHub 操作のみ（commit、PR、status） | Agent ツール | 分離されたサブエージェントでメインコンテキストをクリーンに保つ |
| AskUserQuestion が必要なオーケストレーション | Skill ツール | ユーザーとのやりとりはメインコンテキストが必要 |

## 判断フロー

```
コンテキスト消費量が多い（多数ファイル / Web / 200行超）か？
  はい → Agent ツール（サブエージェント分離）
  いいえ → タスクにプロジェクト固有ルールが必要か？
              はい → Skill ツール（メインコンテキスト）
              いいえ → Agent ツール（分離を優先）
```

## 具体例

| タスク | ルート | 理由 |
|--------|-------|------|
| 機能実装（5件以上のファイル編集） | Agent（`coding-worker`） | ファイル I/O 量 |
| ベストプラクティス調査 | Agent（`research-worker`） | Web アクセス + 大きな出力 |
| 設定品質レビュー | Skill（`reviewing-claude-config`） | `.shirokuma/rules/` へのアクセスが必要 |
| コミット作成 | Agent（`commit-worker`） | git 操作、分離 |
| ルール進化（影響評価） | Agent(Explore)（ステップ 3） | 多数ファイルの読み取り |
| コードレビュー（PR） | Agent（`review-worker`） | 多数ファイル読み取り + 大きなレビュー出力 |
| Issue コメント投稿 | Bash（インライン） | 単一 CLI 呼び出し、最小コンテキスト |
