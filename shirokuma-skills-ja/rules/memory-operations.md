# メモリ運用ルール

## 原則

**メモリはポインタとワーキングコンテキストのみ。**

## メモリに書くもの

| 種類 | 例 |
|------|------|
| GitHub への参照ポインタ | `Discussion #343 参照` `Issue #342` |
| ワーキングコンテキスト | テストコマンド、スキル数などの速引き |

## メモリに書かないもの → GitHub へ

| 種類 | 置き場所 |
|------|---------|
| 調査結果 | Discussion (Research) |
| アイディア・構想 | Discussion |
| 設計決定 | Discussion (ADR) |
| 実装タスク | Issue |
| パターン・知見 | Discussion (Knowledge) → Rule 抽出 |

詳細は `discussions-usage` ルール参照。

## 制約

1. **CLAUDE.md やルールと重複する情報はメモリに書かない** — 情報源は一箇所に
2. **トピック別ファイルも簡素に** — GitHub 参照ポインタのみ

## 設定ファイル内の指示語

設定ファイルは全て同じコンテキストに並列で読み込まれる。相対指示語を使うとどのファイルの指示か区別できない。

| NG | OK |
|----|----|
| 「ここ」「この」「このファイル」 | ファイル名やセクション名で明示 |
| 「上記のルール」 | `branch-workflow ルール` |

## ワークアラウンドはプラグイン不備シグナル

メモリにワークアラウンド（「〜の場合は手動で〜する」等）が残っている場合、それはプラグイン（スキル・ルール・エージェント）の不備を示すシグナルである。メモリは一時的な記録であり、恒久的なワークアラウンド置き場にしない。

**適用方法**: フィードバック系メモリを作成する際、それがプラグインで解決すべき問題なら、Evolution シグナルとして記録し、改善提案につなげる。

## メモリ蓄積 → Evolution フロー

メモリに記憶するものは、将来的にエージェントやスキルの更新に利用することを前提にする。メモリが蓄積してきたら、`evolving-rules` スキルで分析・改善提案につなげる。

**フロー**: メモリ保存 → 蓄積を検知 → Evolution シグナルとして記録 → `evolving-rules` で分析・提案 → スキル・ルールに昇格してメモリ削除。

## 棚卸し判断基準

メモリは極力空にすることがベスト。運用しながら継続調査が必要なものだけ残す。定期的に既存メモリを以下のフローで棚卸しする。

Evolution フローが新規蓄積の継続的トリアージ（メモリ作成時に昇格候補を検知）を担うのに対し、棚卸しは蓄積したメモリ全体を定期的に 4 分類で見直すオフラインの作業として位置付ける。

### 4 分類

| 分類 | 判定基準 | アクション |
|------|---------|----------|
| プラグイン側ルールに昇格 | 全プロジェクトで有効な汎用ルール | `plugin/shirokuma-skills-{ja,en}/rules/` に追記して削除 |
| プロジェクト固有ルールに昇格 | このリポの内部仕様・運用に紐付く | `.shirokuma/rules/{project}/` または `.claude/rules/` に追記して削除 |
| 削除 | コードで強制済み / 完了済み作業の記録 | ファイルと `MEMORY.md` エントリを削除 |
| 保持 | 継続調査中の未解決課題 | 「継続調査中」と識別可能な名前で維持 |

### プラグイン vs プロジェクト固有の判断基準

| 条件 | 昇格先 |
|------|--------|
| shirokuma-docs CLI を使うすべてのプロジェクトで有効 | プラグイン側 |
| プラグインのスキル/ワークフロー自体に関する規約 | プラグイン側 |
| Claude Code 全体のふるまい原則（調査・レビューの進め方等） | プラグイン側 |
| このリポの内部アーキテクチャ・仕様・CLI サブコマンドに紐付く | プロジェクト固有（`.shirokuma/rules/{project}/`） |
| このリポの開発環境（scripts・設定）に固有 | プロジェクト固有（`.claude/rules/`） |
| プロジェクト固有の意思決定・制約（例: pre-release） | プロジェクト固有 |

### 削除判定

以下のいずれかに該当するメモリは削除候補。

- コード/ルール/Hook で既に強制されており、ドキュメントとしても正本が別にある
- 特定 Issue/PR の完了済み作業の実装パターン記録（git log が正本）
- 一度限りの調査結果（Discussion に保存済みか、不要になった）
- 「メモリにある = 今も有効」が成立しない時系列スナップショット

### 棚卸しタスクの進め方

棚卸しは調査タスクとしてメインコンテキストで実行する。サブエージェント丸投げ禁止（全メモリを自分で読んで分類する）。プラグイン側・プロジェクト固有のいずれに昇格するかは上記の判断基準で厳格に切り分ける。
